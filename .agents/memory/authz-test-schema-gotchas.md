---
name: Authorization test suite gotchas (api-server)
description: How the api-server authz regression suite is wired, its test DB isolation, and non-obvious zod-schema pitfalls.
---

The `artifacts/api-server` authorization regression suite (`pnpm --filter @workspace/api-server run test`, vitest + supertest) exercises real routes against a **dedicated `heliumdb_test` database** (isolated from the shared dev `heliumdb`). The pretest drops and recreates `heliumdb_test`, then pushes the drizzle schema into it. Tests create uniquely-emailed throwaway users per run via the real `/auth/register`/`/auth/login` endpoints (promoting to admin via a direct DB update + re-login, since role can't be set through the API).

**Test DB isolation wiring:**
- `lib/db/src/index.ts`: when `NODE_ENV=test` (vitest sets this automatically), `connectionString` is rewritten to `heliumdb_test` via `testDatabaseUrl()` from `lib/db/src/test-db-url.ts`.
- `lib/db/src/setup-test-schema.ts`: drop/recreate `heliumdb_test` (run by `pnpm --filter @workspace/db run setup-test-schema`).
- `lib/db/drizzle.test.config.ts`: drizzle-kit config that targets `heliumdb_test`; used by `push-force-test` script.
- `artifacts/api-server/package.json` pretest: `setup-test-schema && push-force-test`.

**Why `schemaFilter` was needed for drizzle-kit:** using `search_path=test` in the URL without `schemaFilter: ["test"]` causes drizzle-kit 0.31 to silently drop the `test` schema after "Changes applied". Adding `schemaFilter: ["test"]` was tried but also had issues; the separate-database approach (`heliumdb_test`) is the reliable solution.

**Pre-existing test failures (48 tests):** tests in `marketplace.test.ts`, `digital-products*.test.ts`, `negative-price.test.ts`, `fractional-cents.test.ts` fail with 500 because `createDigitalProductStripeCatalog` calls the live Stripe API and fails. This failure exists against both the dev DB and the test DB — it is not caused by the isolation change.

**Orval schema gotcha (still applies):** several `Update*Body` zod schemas generated by orval only accept a subset of fields (e.g. `UpdateDriverBody` only has `status`). Because these schemas aren't `.strict()`, sending an unrecognized field silently parses to an empty update, then causes a 500 from an empty SQL `SET` clause rather than a 400.

**How to apply:** when writing new tests or PATCH payloads, check the actual `Update*Body` zod schema in `lib/api-zod/src/generated/api.ts` for the accepted field names. Also remember tests run against `heliumdb_test`, not `heliumdb`.
