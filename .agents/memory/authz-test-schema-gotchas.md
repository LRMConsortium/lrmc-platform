---
name: Authorization test suite gotchas (api-server)
description: How the api-server authz regression suite is wired and non-obvious zod-schema pitfalls it hit.
---

The `artifacts/api-server` authorization regression suite (`pnpm --filter @workspace/api-server run test`, vitest + supertest) exercises real routes against the actual dev `DATABASE_URL` — there is no separate test database in this project. Tests create uniquely-emailed throwaway users per run via the real `/auth/register`/`/auth/login` endpoints (promoting to admin via a direct DB update + re-login, since role can't be set through the API).

**Why it matters:** several `Update*Body` zod schemas generated by orval only accept a subset of fields (e.g. `UpdateDriverBody` only has `status`, not `vehicleInfo`; `UpdateConstructionProjectBody` only has `status`/`budgetCents`, not `title`). Because these schemas aren't `.strict()`, sending an unrecognized field (like `title`) silently parses to an empty update object instead of failing validation — which then causes a 500 from an empty SQL `SET` clause rather than a 400. This is easy to misdiagnose as an authz bug.

**How to apply:** when writing new tests (or new PATCH payloads) against these routes, check the actual `Update*Body` zod schema in `lib/api-zod/src/generated/api.ts` for the accepted field names before asserting on response bodies — don't assume PATCH accepts the same fields as POST.
